1. Purpose

This service consumes low-confidence chatbot escalation events from RabbitMQ and creates Jira support tickets. It enriches tickets with conversation context, records every Jira attempt, prevents duplicate ticket creation, and exposes operational metrics.

Boundary: the chatbot decides when escalation is needed. This service does not run retrieval, generate answers, or decide confidence thresholds.

3. Event Processing Flow

  1. EscalationEventConsumer receives a raw RabbitMQ message and increments the consumed metric.
  2. Jackson deserializes the JSON into SupportEscalationRequestedEvent.
  3. EventValidator checks required fields, event type, and confidence range.
  4. IdempotencyService checks whether eventId already exists in processed_events.
  5. EscalationService atomically claims the escalation row by updating it to PROCESSING.
  6. ConversationContextService loads the newest 20 conversation messages and returns them chronologically.
  7. JiraIssueBuilder produces the Jira fields payload.
  8. JiraClient calls Jira and classifies errors.
  9. Success updates the escalation to TICKET_CREATED, records the audit attempt, and inserts the processed event.
  10. Failures are recorded, the escalation becomes FAILED, and retryability determines queue behavior.

Database operations use short transactional service methods. The outbound Jira request is not wrapped in a database transaction.

4. Idempotency and Concurrency

The implementation protects the “at most one Jira ticket per escalation” invariant at three layers:

ProtectionImplementation
Repeated event IDprocessed_events.event_id is the primary key and is checked before processing.
Existing Jira ticketThe atomic claim requires jira_ticket_key IS NULL.
Concurrent workersThe atomic update only claims rows in PENDING, QUEUED, or FAILED.
UPDATE escalations
SET status = 'PROCESSING', updated_at = now()
WHERE id = :id
  AND status IN ('PENDING', 'QUEUED', 'FAILED')
  AND jira_ticket_key IS NULL;

Dead-letter queuesupport.escalations.jira.dlq
Dead-letter routing keysupport.escalation.dead-letter

Malformed JSON, invalid events, and non-retryable Jira failures are rejected without requeue. Other failures bubble through a stateless listener retry interceptor with three attempts before rejection and dead-letter routing.

6. Jira Integration

JiraClient uses a basic-auth WebClient and posts to /rest/api/3/issue. The browser URL is derived as {baseUrl}/browse/{key}.

CategoryBehavior
RetryableHTTP 429, all 5xx responses, and WebClient network failures.
Non-retryableOther Jira HTTP errors, including validation and authorization failures.
Resilience4jjiraApi retry: three attempts, 500 ms delay. Circuit breaker: sliding window 10, minimum calls 5.

The issue builder creates a truncated summary, detailed plain-text description, support labels, optional sanitized product label, and priority. Priority is High when priorityHint=high or confidence is below 0.30; otherwise it is Medium.

10. Deployment and Verification

docker-compose.yml starts PostgreSQL 16, RabbitMQ with management UI, and the Jira service on port 8082. The Dockerfile builds a Java 21 runnable jar in a Maven stage and runs it with a Temurin 21 JRE.

Verified locally: Maven compilation, runnable jar packaging, 12 executable unit tests, and four Docker-backed integration scenarios present for PostgreSQL, RabbitMQ, and WireMock. The container scenarios require Docker Desktop.

11. Current Implementation Limits

  • The Jira description is plain text. Atlassian Document Format is not implemented.
  • The listener retry interceptor retries uncategorized runtime failures as well as retryable Jira failures before dead-lettering.
  • The audit attempt number uses count + 1; concurrent retry recording is not guarded by a unique database constraint.
  • Invalid events are rejected to the DLQ but are not persisted with INVALID_EVENT status when deserialization prevents a usable escalation ID.
  • The service expects the chatbot service to create the escalation row before queue publication.
  • Docker Compose points directly at Jira unless JIRA_BASE_URL is overridden; a local WireMock container is not included.
